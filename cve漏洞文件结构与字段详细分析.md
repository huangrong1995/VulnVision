# CVE 漏洞文件结构与字段详细分析

本文档基于用户提供的 `finitestate-cves.json` 文件进行分析，文件内容为固件安全扫描平台导出的漏洞数据。

------

# 1. 文件整体结构

该文件为标准 JSON 数组结构：

```
[    { 漏洞对象1 },    { 漏洞对象2 },    { 漏洞对象3 }]
```

每一个数组元素代表一个独立的漏洞发现（Finding）。

漏洞来源包括：

-  CVE 漏洞情报（SIP_ENRICHMENT） 
-  二进制静态分析（SAST） 
-  固件组件漏洞关联 
-  第三方组件漏洞 

------

# 2. 顶层字段结构

单个漏洞对象整体结构如下：

```
{    "component": {},    "cwes": [],    "description": "",    "detected": "",    "epssPercentile": 0.0,    "epssScore": 0.0,    "epssWeightedRisk": 0.0,    "epssWeightedSeverity": "",    "exploitMaturity": "",    "cveReferences": [],    "comments": "",    "reason": "",    "dataSources": [],    "originalDataSource": "",    "propagated": false,    "exploitInfo": [],    "findingId": "",    "id": "",    "inKev": false,    "inVcKev": false,    "location": "",    "risk": 0,    "severity": "",    "status": "",    "justification": "",    "response": "",    "title": "",    "type": "",    "tracker": "",    "violations": 0,    "warnings": 0,    "vulnInDataset": null,    "reachabilityScore": null,    "attackVector": "",    "factors": null,    "remediation": "",    "mitigation": "",    "advisories": null,    "project": {},    "projectVersion": {}}
```

------

# 3. 字段详细分析

------

# 3.1 component（漏洞所属组件）

表示漏洞所属的软件组件信息。

结构：

```
"component": {    "appId": "",    "id": "",    "name": "",    "vcId": "",    "version": ""}
```

字段说明：

| 字段    | 类型   | 说明                 |
| ------- | ------ | -------------------- |
| appId   | string | 应用/项目ID          |
| id      | string | 组件唯一ID           |
| name    | string | 组件名称             |
| vcId    | string | Version Component ID |
| version | string | 组件版本             |

示例：

```
"component": {    "name": "Linux",    "version": "5.15.170-android13-8-gd2aa3e1c7ab5"}
```

说明：

该漏洞属于 Linux Kernel 5.15.170。

------

# 3.2 cwes（CWE 弱点分类）

Common Weakness Enumeration。

类型：

```
"cwes": [    "CWE-416"]
```

常见 CWE：

| CWE     | 含义                                 |
| ------- | ------------------------------------ |
| CWE-416 | Use After Free                       |
| CWE-476 | NULL Pointer Dereference             |
| CWE-120 | Buffer Overflow                      |
| CWE-787 | Out-of-bounds Write                  |
| CWE-401 | Memory Leak                          |
| CWE-522 | Insufficiently Protected Credentials |

作用：

用于描述漏洞根因。

------

# 3.3 description（漏洞描述）

```
"description": null
```

说明：

漏洞详细描述。

当前文件中多数为空。

通常来源：

-  NVD 
-  MITRE 
-  安全厂商情报 

------

# 3.4 detected（发现时间）

格式：

```
"detected": "2026-05-08T06:39:17Z"
```

说明：

漏洞被扫描系统发现的时间。

格式：

ISO8601 UTC 时间。

------

# 3.5 EPSS 相关字段

EPSS = Exploit Prediction Scoring System

用于预测漏洞被利用概率。

------

## 3.5.1 epssScore

```
"epssScore": 0.00061
```

说明：

漏洞被利用概率。

范围：

```
0.0 ~ 1.0
```

越高风险越高。

------

## 3.5.2 epssPercentile

```
"epssPercentile": 0.19148
```

说明：

漏洞在全部漏洞中的风险百分位。

例如：

```
0.19 = 超过19%的漏洞
```

------

## 3.5.3 epssWeightedRisk

```
"epssWeightedRisk": 0.9
```

说明：

平台综合 EPSS 与 Severity 后计算出的风险值。

------

## 3.5.4 epssWeightedSeverity

```
"epssWeightedSeverity": "LOW"
```

说明：

EPSS 加权后的等级。

常见值：

-  LOW 
-  MEDIUM 
-  HIGH 
-  CRITICAL 

------

# 3.6 exploitMaturity（漏洞利用成熟度）

示例：

```
"exploitMaturity": "poc"
```

说明：

漏洞利用代码成熟度。

常见值：

| 值         | 含义         |
| ---------- | ------------ |
| poc        | 已有 PoC     |
| weaponized | 已武器化     |
| exploited  | 已被真实利用 |

------

# 3.7 cveReferences（漏洞参考链接）

示例：

```
"cveReferences": [    "https://nvd.nist.gov/vuln/detail/CVE-2025-40254"]
```

说明：

漏洞官方参考链接。

通常：

-  NVD 
-  MITRE 
-  Vendor Advisory 

------

# 3.8 dataSources（数据来源）

示例：

```
"dataSources": [    "SIP_ENRICHMENT"]
```

常见值：

| 值              | 含义         |
| --------------- | ------------ |
| SIP_ENRICHMENT  | 漏洞情报增强 |
| SAST            | 静态代码分析 |
| SBOM            | 软件物料清单 |
| Binary Analysis | 二进制分析   |

------

# 3.9 originalDataSource（原始来源）

```
"originalDataSource": "SAST"
```

说明：

漏洞最初来源。

------

# 3.10 findingId（漏洞编号）

示例：

```
"findingId": "CVE-2025-40254"
```

或者：

```
"findingId": "FS-500-009"
```

说明：

漏洞唯一标识。

可能是：

-  CVE 编号 
-  平台内部规则编号 

------

# 3.11 inKev（是否进入 KEV）

Known Exploited Vulnerabilities。

```
"inKev": false
```

说明：

是否进入 CISA KEV 列表。

若为 true：

说明漏洞已被真实攻击利用。

------

# 3.12 risk（综合风险值）

示例：

```
"risk": 78
```

说明：

平台内部风险评分。

通常范围：

```
0~100
```

风险越高越危险。

------

# 3.13 severity（严重等级）

示例：

```
"severity": "high"
```

常见值：

| 值       | 含义 |
| -------- | ---- |
| low      | 低危 |
| medium   | 中危 |
| high     | 高危 |
| critical | 严重 |

------

# 3.14 attackVector（攻击向量）

示例：

```
"attackVector": "NETWORK"
```

常见值：

| 值       | 含义         |
| -------- | ------------ |
| NETWORK  | 网络远程攻击 |
| LOCAL    | 本地攻击     |
| PHYSICAL | 物理接触攻击 |

风险排序：

```
NETWORK > LOCAL > PHYSICAL
```

------

# 3.15 remediation（修复建议）

示例：

```
"remediation": "Replace this certificate with a certificate signed from a trusted Certificate Authority (CA)."
```

说明：

官方修复建议。

------

# 3.16 type（漏洞类型）

示例：

```
"type": "cve"
```

或者：

```
"type": "binary-sast"
```

常见值：

| 值          | 含义               |
| ----------- | ------------------ |
| cve         | CVE漏洞            |
| binary-sast | 二进制静态分析漏洞 |
| sast        | 静态代码扫描       |
| dependency  | 依赖漏洞           |

------

# 3.17 violations / warnings

示例：

```
"violations": 1,"warnings": 0
```

说明：

平台规则触发统计。

------

# 3.18 project（所属项目）

结构：

```
"project": {    "id": "",    "name": ""}
```

说明：

漏洞所属项目。

示例：

```
"project": {    "name": "ORION-M-V4"}
```

------

# 3.19 projectVersion（项目版本）

结构：

```
"projectVersion": {    "id": "",    "version": "",    "created": "",    "updated": ""}
```

说明：

漏洞对应的固件版本。

------

# 4. 漏洞类型分析

该文件中包含两大类漏洞：

------

# 4.1 CVE 漏洞

特点：

-  有 CVE 编号 
-  来源于漏洞数据库 
-  带 NVD 引用 
-  带 EPSS 风险 

示例：

```
"type": "cve"
```

------

# 4.2 Binary SAST 漏洞

特点：

-  来源于二进制静态分析 
-  无 CVE 
-  属于代码质量/安全问题 

示例：

```
"type": "binary-sast"
```

包括：

-  自签名证书 
-  未检查返回值 
-  权限错误 
-  不安全证书 

------

# 5. 风险字段关系分析

风险评估通常来自：

```
CVSS+ EPSS+ Exploitability+ Attack Vector+ Reachability+ Severity
```

综合形成：

```
"risk": 78
```

------

# 6. 典型高风险漏洞案例

------

## 6.1 CVE-2025-21927

```
"severity": "high","risk": 78,"attackVector": "LOCAL","cwes": ["CWE-787"]
```

分析：

-  越界写 
-  高危内存破坏 
-  可导致提权/崩溃 

------

## 6.2 CVE-2025-0725（cURL）

```
"exploitMaturity": "poc","attackVector": "NETWORK"
```

分析：

-  已存在 PoC 
-  网络可攻击 
-  风险较高 

------

## 6.3 PEM 证书问题

```
"type": "binary-sast","severity": "critical"
```

分析：

-  固件中包含 PEM 证书 
-  可能泄露私钥 
-  属于供应链风险 

------

# 7. 文件用途分析

该文件通常用于：

| 场景         | 用途            |
| ------------ | --------------- |
| 固件安全分析 | IoT设备漏洞识别 |
| SBOM安全审计 | 第三方组件风险  |
| DevSecOps    | CI/CD安全门禁   |
| 漏洞治理     | 修复追踪        |
| 风险评分     | 安全报告生成    |

------

# 8. 推荐后续处理流程

建议流程：

```
JSON漏洞文件    ↓漏洞归类    ↓风险排序    ↓生成修复建议    ↓生成HTML/PDF报告    ↓漏洞跟踪闭环
```

------

# 9. 推荐的数据建模

建议转换为如下结构：

```
{  "cveId": "",  "component": "",  "version": "",  "severity": "",  "risk": 0,  "attackVector": "",  "cwe": [],  "references": [],  "remediation": ""}
```

方便：

-  Elasticsearch 
-  PostgreSQL 
-  Neo4j 
-  风险聚合分析 

------

# 10. 总结

该 CVE 文件本质是：

```
固件/软件漏洞扫描结果集合
```

融合了：

-  CVE情报 
-  EPSS风险 
-  SAST结果 
-  固件组件信息 
-  项目版本信息 

其核心价值：

```
帮助企业识别：“哪些组件存在漏洞”“漏洞是否可利用”“风险有多高”“如何修复”
```
